# Codex 협업 지침

## 1. 프로젝트 개요

- 이 저장소는 해커톤에서 친구와 함께 개발하는 Unity 2D 협동 게임 프로젝트다.
- 게임 기획의 기준 문서는 `Codex/GAME_DESIGN.md`다.
- Unity 버전은 `6000.3.15f1`이며 Universal 2D 템플릿을 사용한다.
- 두 작업자 모두 Codex를 사용하므로 다른 작업자의 변경 사항이 언제든 존재할 수 있다.

## 2. 작업 시작 전 필수 확인

1. `Codex/GAME_DESIGN.md`와 이 문서를 읽는다.
2. `git status --short`로 기존 변경 사항을 확인한다.
3. 최근 작업 기록 중 현재 작업과 관련된 내용을 확인한다.
4. 사용자 소유의 변경 사항을 덮어쓰거나 되돌리지 않는다.
5. 요구사항이 미확정 기획과 충돌하면 임의로 확정하지 말고 가정 또는 제안임을 명시한다.

## 3. 작업 기록 규칙

- Codex가 저장소를 변경한 매 작업마다 `Codex/WorkLogs/` 아래에 작업 기록 Markdown 파일을 만든다.
- 모든 작업 기록의 파일명에는 대소문자를 그대로 지킨 `Leewonwoo`를 반드시 포함한다.
- 권장 파일명 형식은 `YYYY-MM-DD_HHMM_Leewonwoo_<작업명>.md`다.
- 같은 분에 파일명이 겹치면 뒤에 `-02`, `-03`처럼 순번을 붙인다.
- 작업 기록에는 최소한 다음 내용을 남긴다.
  - 작업 목적
  - 변경한 파일
  - 핵심 변경 내용
  - 확인 또는 테스트 결과
  - 남은 작업이나 주의 사항
- 문서, 코드, 설정을 실제로 변경하지 않은 단순 질의응답은 작업 기록 대상에서 제외한다.
- 모든 Markdown 문서는 `Codex/` 폴더 안에 둔다.

## 4. Git 협업 규칙

- 작업 전후로 `git status --short`를 확인한다.
- 다른 작업자의 변경을 임의로 삭제, 복원, 스테이징 또는 커밋하지 않는다.
- 커밋 요청을 받으면 이번 작업에서 직접 변경한 파일만 경로를 지정해 스테이징한다.
- 사용자가 요청하지 않는 한 `push`, 강제 푸시, 브랜치 삭제, 히스토리 재작성은 하지 않는다.
- `git reset --hard`, 무단 `git checkout -- <file>` 등 파괴적인 명령을 사용하지 않는다.
- 하나의 커밋에는 가능한 한 하나의 목적만 담고, 커밋 메시지는 변경 의도가 드러나게 작성한다.
- Unity 씬, 프리팹, ProjectSettings처럼 충돌 가능성이 큰 파일은 동시에 수정하지 않도록 먼저 역할을 나눈다.

## 5. Unity 프로젝트 규칙

- `Library/`, `Temp/`, `Logs/`, `UserSettings/` 등 Unity 생성 폴더를 커밋하지 않는다.
- 에셋 파일을 추가하거나 이동할 때 대응하는 `.meta` 파일도 함께 관리한다.
- Unity 에디터 밖에서 에셋을 이동할 경우 파일과 `.meta` 파일을 함께 이동한다.
- 씬 전체에 직접 결합하기보다 재사용 가능한 기능은 컴포넌트와 프리팹으로 분리한다.
- 새 입력 기능은 현재 설치된 Input System 패키지를 기준으로 구현한다.
- 물리 처리는 가능한 한 `FixedUpdate`, 입력 수집은 `Update` 또는 Input System 콜백 등 Unity 생명주기에 맞게 분리한다.
- 공개 필드는 꼭 필요한 경우에만 사용하고, Inspector 노출은 기본적으로 `[SerializeField] private`을 사용한다.
- 코드와 파일 이름은 영문을 사용하되, 사용자에게 보이는 텍스트와 문서는 한국어를 사용할 수 있다.

## 6. 구현 원칙

- 해커톤 일정에 맞춰 가장 작은 플레이 가능한 흐름을 먼저 완성한다.
- 기획서에서 `미확정` 또는 `제안`으로 표시된 내용을 확정 요구사항처럼 구현하지 않는다.
- 토끼와 거북이 중 한쪽만으로 퍼즐이 해결되지 않도록 협동의 필요성을 유지한다.
- 시점 전환 중 게임은 정지하지 않고 슬로 모션으로 진행된다는 핵심 규칙을 보존한다.
- 토끼의 속도나 관성이 거북이에게 전달되는 기능으로 잘못 해석하지 않는다.
- 이로운 효과를 얻은 대가로 다음 스테이지에 역경이 추가되는 구조를 유지한다.

## 7. 검증과 완료 기준

- 변경 범위에 맞는 가장 작은 검증부터 실행한다.
- C# 코드 변경 시 가능하면 컴파일 오류 여부를 확인한다.
- 플레이 동작 변경 시 확인해야 할 Unity 에디터 재현 절차를 작업 기록에 남긴다.
- 자동 검증을 실행하지 못했다면 실행하지 않았다는 사실과 필요한 수동 확인을 명시한다.
- 완료 보고에는 변경 요약, 검증 결과, 남은 위험 요소를 간결하게 포함한다.

## 8. 문서 관리

- 확정된 기획 변경은 `Codex/GAME_DESIGN.md`에 반영한다.
- 아이디어와 확정 사항을 섞지 말고 상태를 명확히 표시한다.
- 중복 문서를 만들기보다 기존 기준 문서를 갱신한다.
- 새로운 Markdown 파일은 모두 `Codex/` 아래에 생성한다.
